1. Vision General del Sistema

El Sistema de Gestion de Citas de Bienestar SENA es una aplicacion web SPA construida con React que permite gestionar citas entre aprendices del SENA y profesionales de bienestar.

Actores del Sistema

RolDescripcionPermisos
SUPERADMINAdministrador del sistemaCRUD completo de usuarios, roles, dependencias, configuracion, auditoria.
COORDINACIONCoordinador de BienestarSupervision de citas, reportes consolidados, vista multi-dependencia.
PROFESIONALPsicologia / Enfermeria / Trabajo SocialGestion de citas de su dependencia, historial de aprendices.
APRENDIZAprendiz del SENASolicitar citas, ver/cancelar sus propias citas.

2. Arquitectura

2.1 Diagrama de Arquitectura

+-----------------------------------------------------------------------+ | CAPA DE PRESENTACION | | React 19 + React Router 7 + Lucide React + Recharts | | | | [Login] [Dashboard Aprendiz] [Admin Panel] [Reports Panel] ... | +-----------------------------------------------------------------------+ | CAPA DE LOGICA | | Custom Hooks: useAppointments, useAdmin, useReports | | React Hook Form + Zod (validacion) | | AuthProvider (context + RBAC) | +-----------------------------------------------------------------------+ | CAPA DE ACCESO A DATOS | | Repository Pattern: | | [AppointmentRepo] [AdminRepository] [DashboardRepo] | +-----------------------------------------------------------------------+ | CAPA DE SERVICIOS | | Supabase JS Client v2 | | [ Supabase Cloud (PostgreSQL + Auth + Realtime + RLS) ] | +-----------------------------------------------------------------------+

2.2 Patrones de Arquitectura

  • Separacion de Concerns: Cada capa tiene una responsabilidad unica.
  • Repository Pattern: El acceso a datos esta encapsulado en clases repositorio.
  • Custom Hooks: La logica de negocio se extrae en hooks reutilizables.
  • Feature-Based Organization: Codigo organizado por dominio (appointments, admin, dashboard, auth).

2.3 Flujo de Datos

Componente --> Custom Hook --> Repository --> Supabase Client --> Supabase API
    |              |              |              |
    +-- Render <-- State <-- Data <-- Promise <-- Response

5.3 Row Level Security (RLS)

-- Aprendiz solo ve sus propias citas
CREATE POLICY "Aprendices ven sus citas"
ON appointments FOR SELECT
USING (
  user_id = auth.uid()
  OR professional_id = auth.uid()
  OR EXISTS (
    SELECT 1 FROM profiles
    WHERE id = auth.uid()
    AND role_id IN (1, 2)  -- SUPERADMIN, COORDINACION
  )
);

6.2 Helpers RBAC en AuthProvider

const hasRole = (roleName) => {
  const role = profile?.roles?.name?.toUpperCase();
  if (Array.isArray(roleName))
    return roleName.some(r => role === r.toUpperCase());
  return role === roleName.toUpperCase();
};

const isAdmin = () => hasRole("SUPERADMIN");
const isCoordination = () => hasRole("COORDINACION") || isAdmin();
const isAprendiz = () => hasRole("APRENDIZ");

11. Validaciones (Zod)

11.1 Schema de Cita

export const appointmentSchema = z.object({
  dependency_id: z.number().int().positive(),
  scheduled_date: z.string()
    .refine(d => !isWeekend(new Date(d)))
    .refine(d => !isPast(startOfDay(new Date(d))))
    .refine(d => new Date(d) >= startOfDay(addDays(new Date(), 1))),
  scheduled_time: z.string()
    .regex(/^[0-9]{2}:[0-9]{2}$/)
    .refine(t => { const h = Number(t.split(":")[0]); return h >= 8 && h < 17; }),
  reason: z.string().min(10).max(500),
  notes: z.string().max(1000).optional(),
});

11.2 Reglas de Validacion

13. Despliegue

13.1 Requisitos

  • Node.js 18+
  • npm 9+
  • Cuenta de Supabase con proyecto configurado

13.2 Instalacion

# Clonar
git clone https://github.com/[org]/gestion-citas.git
cd gestion-citas

# Instalar dependencias
npm install

# Configurar variables de entorno
cp .env.example .env
# Editar .env con credenciales de Supabase

# Ejecutar en desarrollo
npm run dev

# Build para produccion
npm run build
